# MAGI System Update Policy

## 1. このアプリの目的

MAGI System は、AI に判断を委任するアプリではない。異なる価値観をぶつけて、盲点、反対理由、実行条件を露出させるための意思決定支援 UI である。

目的は次の 3 点に集約する。

- 対立を見せる
- 条件付きの意思決定を見せる
- 人間が最終判断しやすい材料を整える

## 2. 何を優先するか

今後の更新では、次の優先順位を守る。

1. 審議が割れること
2. 割れた理由が読めること
3. 合議結果から次の行動が分かること
4. そのうえで見た目が強いこと

見栄えだけが増えて、審議の差分が弱くなる変更は避ける。

## 3. パネル設計の原則

### BALTHASAR

- 役割: 前提破壊
- 主に問うもの: 常識依存、楽観、言い換えだけの改善、権威への依存
- 承認しやすい条件: 挑戦する価値が高く、失敗しても引き返せる

### CASPAR

- 役割: 被害監査
- 主に問うもの: 権利侵害、悪用、炎上、弱者へのしわ寄せ、説明責任
- 承認しやすい条件: 保護策、監視体制、責任の所在が具体的

### MELCHIOR

- 役割: 執行責任
- 主に問うもの: 実装可能性、責任者、コスト、運用負荷、撤退計画
- 承認しやすい条件: 小さく試せて、失敗時に止められる

### 禁止事項

- 3 パネルを似た性格に寄せること
- 中立や要約に逃がして衝突を弱めること
- 結論だけ一致して理由が似る状態を放置すること

## 4. 出力フォーマット契約

各パネルは、最低限次の要素を必ず含める。

- `結論`
- `理由`
- `最大リスク`
- `承認条件`
- `判定`

この順序を変える場合は、`index.html` 内のパース処理と README を同時に更新する。

## 5. 合議 UI の契約

合議エリアでは、最低限次を表示し続ける。

- 投票結果
- 最終判定
- 多数派の要約
- 少数意見
- 保留条件
- 推奨アクション

多数決だけを大きく見せて、少数意見や条件を隠す変更は行わない。

## 6. ゲーム性を高める方針

- 良い問いの例を UI に出し、衝突しやすい入力を促す
- 満場一致より 2 対 1 の緊張感を歓迎する
- 条件付き承認、差し戻し、保留など、結果のグラデーションを維持する
- 「誰が何を嫌がっているか」を読み物として楽しめる構造にする

## 7. ドキュメント更新方針

機能追加や仕様変更のたびに、少なくとも次を確認する。

- `README.md`
- `CHANGELOG.md`
- `docs/UPDATE_POLICY.md`
- 必要なら `CLAUDE.md`

更新が不要な場合も、なぜ不要かを作業報告で短く説明する。

## 8. 非目標

現時点での非目標は以下。

- 厳密な業務意思決定システムとしての保証
- モデル比較ベンチマーク
- 長文レポート生成ツール化
- フレームワーク導入やビルド工程の追加
